# Inductify — 面试故事

## 一句话介绍

一个 AI 驱动的员工入职问答机器人，能回答公司政策类问题并附带来源引用，在 54 题保留评估集上实测准确率达 83%。

---

## 我解决的问题

新员工需要花大量时间在政策文档里翻找基本答案："VPN 配置是什么？""我有多少天 PTO？""报销上限是多少？"

通用 LLM 无法回答这类问题——在公司内部知识上的准确率 **不足 2%**，因为它根本无法访问内部文档。

---

## 我构建的系统

一个全栈 RAG 系统：

```
用户 → Next.js UI → FastAPI 后端 → LangChain RAG 链
                                  ├── ChromaDB（10K+ 文档）
                                  ├── CrossEncoder 重排序器
                                  └── GPT-4.1-mini（带接地提示词）
```

- `POST /ask` — RAG 链，返回答案 + 来源引用
- `POST /agent/ask` — ReAct Agent，使用 `vector_search` + `list_documents` 工具
- `POST /upload` + `POST /index` — 异步文档入库流水线
- 每用户独立会话历史（内存存储，可清除）

---

## 核心工程问题

### 初始尝试：59% 准确率，重排序器无效

我用 54 题合成评估集跑了一次受控消融实验，结果如下：

- 纯 LLM（无检索）：**1.9%**
- RAG k=3：**59.3%**
- RAG k=10 + CrossEncoder 重排：**59.3%** — 完全一样，零提升

我深入排查，发现是**两个独立的 Bug**：

**Bug 1 — 干扰文档污染（主要原因）**

语料库中有 1,000 份"已废弃政策"文档（旧版本，数值不同）。这些文件只有 5–7 行——极短、关键词密度极高。在向量空间中，一篇写着"VPN VPN-5554"的 6 行干扰文档，其 embedding 得分高于写着"VPN：使用配置文件 VPN-2424"的 200 行正式政策文档——因为短文本 embedding 中的噪声接近于零。

CrossEncoder 无法修复这个问题：对于"VPN"查询来说，"VPN-5554"和"VPN-2424"在语义上都是相关的。**语义相关性 ≠ 事实正确性**。

**Bug 2 — 子串匹配过于严格（评估评分 Bug）**

"60 calendar days"不包含子串"60 days"，导致正确答案被判为错误。修复方案：数字限定词正则放宽——`\b60\s+\w+\s+days` 即可匹配预期的"60 days"。

### 解决方案：仅使用政策文档的向量库 + 改进评分

- 从评估库中提取 4,172 条政策文本块（复用已存储的 embedding，零 API 调用），写入独立的向量库
- 用该纯净语料库重新跑评估

最终结果：

| 条件 | 准确率 |
|------|--------|
| 纯 LLM（无检索） | 1.9% |
| RAG，混合语料库 | 64.8% |
| RAG，仅政策文档 | **83.3%** |

**83% = 生产环境数字**：在真实部署中，你只会索引当前有效文档，不会索引旧版本。

---

## 重排序器为什么仍然无效（以及为什么这是正确的）

即使在纯净语料库下，k=3 余弦相似度 = k=10 + 重排的准确率依然相同。

原因：对于 45 道可答题，正确文本块已经在排名 1–3 中。对于 9 道失败题，正确文本块根本不在前 10——重排无法拯救检索阶段就没找到的内容。

但重排序器在生产环境中仍有价值：
- 让来源引用对用户更有意义（排序更合理）
- 在正确文档位于第 4–10 名的场景下能提升准确率（更大、更多样化的语料库中很常见）
- 只把前 3 条传给 LLM，降低 Token 开销

## Agent 层（F9）

在 RAG 链基础上，增加了 ReAct 风格的 Agent（`POST /agent/ask`）：

```python
create_agent(llm, [vector_search, list_documents], system_prompt=...)
```

两个工具：
- `vector_search(query)` — 从 ChromaDB 检索 + 重排序
- `list_documents(query)` — 返回可用的政策分类

Agent 自主决策："我应该检索这个，还是告诉用户我能帮什么？"——这就是"Agentic"行为的体现。固定 RAG 流水线总是执行检索；Agent 可以选择不检索。

---

## 我会怎么改进

1. **从一开始就用 Metadata 过滤索引** — 建索引时给每份文档打上 `doc_type: current | archived` 标签，检索时直接过滤。就不需要维护两套向量库了。
2. **更大的 k + 重排** — k=50 → 重排 → 前 3，能覆盖那 9 道正确文本块不在前 10 的失败题。
3. **BM25 混合检索** — 结合稀疏检索（BM25）和稠密检索（OpenAI embedding）。BM25 对精确代码匹配（如"VPN-2424"）的处理优于余弦相似度。

## 常见追问

**Q：为什么选 ChromaDB 而不是 Pinecone / Weaviate？**

本地部署、零成本、持久化存储在磁盘上、与 LangChain 原生集成。对于作品集项目来说是正确的权衡。生产环境中会根据规模评估托管方案。

**Q：如果文档量增长到 100 万，怎么扩展？**

ChromaDB → Pinecone 或 pgvector。加 Embedding 缓存（Redis），避免相同文本块重复嵌入。换用异步批量 Embedding（OpenAI Batch API，成本降低 50%）。

**Q：重排序器为什么没效果？**

见上文——干扰文档污染是主要问题，而不是排序问题。语料库清理后准确率提升了 18 个百分点，根本没动重排序器。（这是一个很好的调试故事：我最初以为是重排序器的问题，做了诊断后发现了真正的根本原因。）

**Q：会话历史怎么处理？**

用 `session_id`（UUID）为键的内存字典，每个会话存储 `[(human, ai), ...]` 对。生产环境应使用 Redis 或数据库持久化，以保证重启后历史不丢失。`DELETE /session/{id}` 可清除指定会话。

**Q：`/ask` 和 `/agent/ask` 的区别是什么？**

`/ask` 是固定 RAG 流水线：总是执行检索，总是生成答案。`/agent/ask` 使用工具调用 Agent，能自主决定是检索还是直接回答——支持"你能帮我做什么？"这类不需要检索的问题。
